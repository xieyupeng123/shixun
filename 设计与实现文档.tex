\section{面向对象设计}
\subsection{设计原则}
\begin{enumerate}[leftmargin=2em]
    \item \textbf{封装}：每个类管理自己的状态，外部通过方法交互
    \item \textbf{继承}：玩家和敌人继承公共实体基类，复用移动和碰撞
    \item \textbf{多态}：同一移动方法在不同子类中表现不同（键盘驱动 vs AI驱动）
    \item \textbf{组合}：关卡类组合管理玩家、敌人、武器等对象
\end{enumerate}

\subsection{核心类职责}

\subsubsection{实体基类}
封装公共行为：统一移动（速度+方向向量+帧时间）、分轴碰撞检测（先水平后垂直）、Hitbox缩小碰撞框、动画帧管理。

\subsubsection{玩家类}
输入驱动：键盘输入→设置方向→切换动画状态→攻击/魔法触发（通过回调通知关卡层）。冷却管理通过时间戳差值判断。能量每帧自动恢复。

\subsubsection{敌人类}
AI驱动：三态状态机（待机/追踪/攻击），根据距离切换。追踪时调用A*寻路，路径缓存+定时重算。死亡时通知关卡层播放特效、增加经验。

\subsubsection{关卡管理类}
核心协调者：地图解析→实体创建→碰撞调度→传送管理→存档接口。

\section{核心算法分析}

\subsection{寻路算法对比}

本项目需要在网格地图中为敌人找到绕过障碍物到达玩家位置的最短路径。对比了以下四种算法：

\subsubsection{广度优先搜索（BFS）}
\begin{itemize}[leftmargin=2em]
    \item \textbf{原理}：从起点逐层扩展，先访问所有距离为1的节点，再访问距离为2的节点
    \item \textbf{优点}：一定找到最短路径，实现简单
    \item \textbf{缺点}：无方向性，盲目扩展所有方向，搜索范围大
    \item \textbf{复杂度}：$O(V+E)$
    \item \textbf{评价}：保证最短但效率低，不适合实时游戏
\end{itemize}

\subsubsection{Dijkstra算法}
\begin{itemize}[leftmargin=2em]
    \item \textbf{原理}：每次选择当前代价最小的节点扩展，逐步到达终点
    \item \textbf{优点}：保证最短路径，支持加权图
    \item \textbf{缺点}：无启发式引导，搜索范围仍较大
    \item \textbf{复杂度}：$O((V+E)\log V)$（优先队列）
    \item \textbf{评价}：本项目网格权重相同，Dijkstra的加权能力用不上
\end{itemize}

\subsubsection{A*算法（最终选择）}
\begin{itemize}[leftmargin=2em]
    \item \textbf{原理}：$f(n) = g(n) + h(n)$，综合实际代价和启发估计
    \item \textbf{启发函数}：曼哈顿距离 $h(n) = |x_1-x_2| + |y_1-y_2|$
    \item \textbf{优点}：保证最短路径，启发式剪枝大幅减少搜索范围
    \item \textbf{缺点}：最坏情况退化为Dijkstra
    \item \textbf{复杂度}：最坏$O(E)$，实际远小于此
    \item \textbf{评价}：本项目的最佳选择——网格地图、静态障碍物、需最短路径
\end{itemize}

\subsubsection{贪心最佳优先搜索}
\begin{itemize}[leftmargin=2em]
    \item \textbf{原理}：只考虑$h(n)$，每次选离终点最近的节点扩展
    \item \textbf{优点}：搜索速度极快
    \item \textbf{缺点}：不保证最短路径，可能走入死胡同
    \item \textbf{复杂度}：$O(E)$
    \item \textbf{评价}：速度最快但路径质量无保证，不适合本项目
\end{itemize}

\subsection{碰撞检测算法}

采用分轴AABB碰撞检测，将二维碰撞分解为两个一维碰撞：

\begin{enumerate}[leftmargin=2em]
    \item 更新$x$坐标后，检测水平方向碰撞并修正位置
    \item 更新$y$坐标后，检测垂直方向碰撞并修正位置
    \item 修正方式：将碰撞框回退到障碍物边界
\end{enumerate}

\textbf{为什么分轴}：若同时处理两个轴，角色可能在对角线方向同时穿过两个相邻障碍物。分轴处理从根本上避免了这个问题。

\subsection{Y轴排序渲染}

为实现正确遮挡关系（画面下方的角色挡住上方的角色）：

\begin{enumerate}[leftmargin=2em]
    \item 计算相机偏移量（以玩家位置为中心）
    \item 绘制地面贴图
    \item 将所有可见精灵按$y$坐标排序
    \item 按排序顺序依次绘制，$y$值大的后绘制
\end{enumerate}

\subsection{动态规划思想在升级系统中的应用}

升级系统中，玩家需要决定如何分配有限的经验值。这本质上是一个资源分配优化问题：

\begin{itemize}[leftmargin=2em]
    \item \textbf{问题建模}：给定总经验值$E$，五项属性的升级消耗均为$c$，每项属性有上限，求最优分配方案使战斗力最大化
    \item \textbf{贪心策略}：优先升级性价比最高的属性（如攻击力每级+2但消耗100经验，生命值每级+20消耗100经验）
    \item \textbf{动态规划}：若属性间存在协同效应（如魔法属性同时影响魔法伤害和能量回复），则需要考虑组合优化
    \item \textbf{本项目简化}：各属性独立，玩家自由选择，无需AI自动分配
\end{itemize}

\section{数据结构设计}
\subsection{配置数据}
武器、怪物、魔法数据采用字典嵌套结构，数据与逻辑分离（数据驱动设计），新增内容无需修改核心代码。

\subsection{地图数据}
每张地图由四个CSV文件描述，每个文件为二维矩阵。特殊值标识玩家出生点、敌人类型、传送点目标。

\subsection{存档数据}
JSON格式，包含玩家状态快照、已击败敌人坐标列表、已破坏草地坐标列表。

\section{场景交互设计}
\subsection{攻击交互}
按键→创建武器精灵→放置在朝向方向→检测碰撞→敌人扣血→冷却后移除武器。

\subsection{场景切换}
检测传送点→淡出→销毁旧关卡→加载新地图→创建新关卡→淡入。玩家属性跨地图保留。

\subsection{升级交互}
按M暂停→打开菜单→显示属性→选择升级→消耗经验→按M恢复。
