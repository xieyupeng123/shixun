我正在进行一个软件工程课程实训项目——「星露谷战斗模块」（Stardew Valley Action X），采用B模式（AI智能协作开发）。技术栈已确定：Python 3.12 + Pygame 2.6，窗口分辨率1280×720，帧率锁定60FPS，地图数据用CSV格式（不用TMX），存档用JSON。

在让你写任何一行代码之前，我先把我的设计文档全部给你。桌面上的「文档」文件夹里有我提前写好的全套软件工程文档，你要逐一读完：

1. 需求规格说明书（SRS）—— 18个功能需求的完整定义、优先级划分（P0/P1/P2）、非功能需求、验收标准、需求追踪矩阵
2. 概要设计说明书（HLD）—— 四层分层架构、模块划分、面向对象类图、核心状态机设计、数据结构设计
3. 需求文档 —— 更早期的需求讨论和细化
4. 设计与实现文档 —— 架构细节和测试方案

这些文档是我在需求工程阶段花了大量时间打磨出来的，你对项目的所有理解都应该以这些文档为基准。我的代码结构、类的命名、模块的职责划分，都严格遵循这些设计文档，你要按照我的设计思路来，不要自己发挥。

架构硬性要求：
- 采用四层分层架构：控制层 → 逻辑层 → 实体层 → 数据层
- 上层依赖下层，下层绝不依赖上层，同层之间通过回调函数通信
- 每个模块职责单一（SRP原则），我检查代码时会看这个
- Level作为核心协调者，通过组合模式集成CombatManager、BossManager、MapRenderer等专职Manager，不要把所有逻辑堆在Level里
- ResourceManager必须是单例模式，全局统一管理资源加载和缓存
- 武器、怪物、魔法的属性数据全部集中在settings.py的字典中，数据与逻辑分离，新增类型只需加配置不改核心代码
- 所有资源路径使用相对路径，项目可以移植到任何机器直接运行

代码质量要求：
- 面向对象，多文件多模块结构，每个类的职责边界清晰
- 封装要做好，类的内部状态不允许外部直接访问，通过方法暴露接口
- 低耦合，模块之间的依赖尽可能少，Player不应该直接知道Enemy的存在
- 游戏主循环用delta time驱动，不要用固定帧数做动画
- 所有异常要有降级处理，资源缺失不能导致崩溃，图片加载失败用默认占位，音效加载失败静默跳过

功能实现顺序严格按我规划的三阶段来：

第一阶段（核心跑通）：主菜单界面 + 角色移动和12组动画（4方向×3状态：待机/行走/攻击）+ CSV四层地图加载和视窗跟随 + 分轴AABB碰撞检测 + 边界限制

第二阶段（战斗闭环）：鼠标左键武器攻击 + 鼠标右键魔法释放（火焰AOE和治疗回血）+ Q键/E键切换武器/魔法（200ms冷却防误触）+ 五种武器的不同伤害和冷却属性 + 四种怪物的AI状态机（待机→A*寻路追踪→攻击→受击→死亡）+ 粒子特效系统 + 音效系统

第三阶段（系统完善）：Boss三阶段系统（每击杀晋升下一阶段，属性×1→×2→×4，颜色正常→淡红→紫色）+ B键升级菜单（五维属性升降级，经验消耗，上限限制）+ P键存档（JSON原子写入）+ M键世界地图（障碍网格+敌人红点+Boss红叉标记+玩家绿点）+ 空格无敌模式（速度×2攻击×2金色光效免疫伤害）+ 死亡/胜利结算界面 + Boss音乐区域切换（接近Boss自动切BGM）+ 水面像素级蓝色检测

交互和操控——我在需求阶段就和AI讨论确定的：
- 移动：WASD/方向键，对角线速度归一化
- 攻击：鼠标左键，朝向方向生成武器精灵，各武器冷却时间不同
- 魔法：鼠标右键，治疗术回血+光环粒子，火焰术前方5格AOE
- 切换：Q键/滚轮切换武器，E键切换魔法
- B键暂停打开升级菜单（五项属性升降级）
- M键全屏世界地图
- 空格切换无敌模式
- P键在暂停时存档
- 主菜单上下键选、回车确认、ESC退出

每完成一个阶段我会运行测试确认没问题再推进。你每写完一个模块要告诉我：这个模块的职责是什么、对外暴露了哪些接口、依赖了哪些其他模块、有没有违反我上面提的架构原则。

关于AI模型使用策略——我会根据场景切换不同模型：
- 代码架构、类设计、重构等纯逻辑任务 → 用推理能力强的模型
- 构建地图、处理像素级水域检测 → 切换到有图像识别能力的多模态模型（比如MIMO），因为要分析ground.png的蓝色通道来判断水域
- 写文档、整理需求 → 用长文本处理好的模型
- 调试运行时Bug → 先把完整报错堆栈给推理模型分析，再切回代码模型修复
